# Report: Dataset Collection and Label Preprocessing

This report explains how Chinese chess game records are turned into a structured machine-readable corpus and how each move in that corpus is checked and labeled for legality under the project rule engine. The description follows the actual scripts and modules in the Xiangqi AI repository so that documentation matches implementation.

## Data sources and raw game format

The project works from games that are stored in a PGN-style layout: text headers describe the event, players, result, and sometimes a FEN string for a non-standard starting position, followed by a body of move text. In community and historical corpora, these files are not always saved as plain UTF-8. Encodings such as Big5, GB family, and CP950 appear often, so any ingestion step must read bytes and try several decoders instead of failing on the first decode error. The FEN string, when present, fixes the initial board and side to move; the move list that follows is written in Chinese algebraic style as used in the source collection rather than in Western long algebraic form.

## From PGN files to JSON Lines

The parse script under tools/parse_pgn_fen.py is the main entry point for building a single-file dataset that downstream code can stream line by line. It accepts either one PGN file or a directory tree of PGN files. For each file it reads the raw bytes and attempts decoding in a fixed order (UTF-8, UTF-8 with BOM, CP950, Big5, GB18030, GBK) until one encoding succeeds, and it records which encoding was used so that provenance is clear. Header lines are recognized by the usual bracket pattern and stored as key-value metadata. The move section is treated as free text: parenthetic comments and braced comments are removed, result tokens and move numbers are dropped, and the remaining tokens are kept in order as a list of move strings. Each completed game becomes one line in a JSONL file, that is one JSON object per line. The object includes the relative path to the source PGN, the successful encoding name, the header map, the FEN if present, and the list of move strings. A file such as data/ccpd_parsed.jsonl in the project is exactly this form: the heavy lifting of parsing PGN and splitting moves is already done, and the rest of the pipeline works on JSONL only.

## Why notation must be normalized

Move text in the corpus mixes full-width and half-width digits, Chinese number characters, and piece names that may appear in traditional or simplified forms. The engine, however, reasons over internal board states and legal move objects, not over raw strings. The module tools/validate_pgn_legality.py therefore implements a function that normalizes one move token into a single comparable string. The normalization maps piece names to a consistent set, turns various digit conventions into ordinary ASCII digits, and removes decorative characters such as extra spaces or move-number prefixes that are not part of the move itself. After normalization, the code can ask which legal moves exist in the current position and ask, for each legal move, which string aliases describe that move in human notation. The function that enumerates those string forms is aligned with the same board geometry and side-to-move rules as the rest of the engine. Matching tries the usual file-based Chinese convention first; if the token does not match uniquely, a second pass may use an absolute file convention that some databases use for both colors. The training dataloader in models/dataset.py replays games with this same matching strategy so that training data and legality checks never disagree on what a line of text is supposed to mean.

## Generating per-move legality labels

A separate pass over the parsed JSONL adds explicit supervision-friendly metadata. The script tools/label_moves_legality.py reads each line as a JSON object, loads the FEN into a game state with the same from_fen routine used elsewhere, and then walks the move list in order. If the FEN cannot be built, every move in that record is marked as not legal and the reason records an invalid FEN. For a valid start, each move string is normalized. An empty token after normalization is labeled with a dedicated reason so that blank or garbage entries are visible in the output. When the token is non-empty, the script finds the unique legal move that matches the token, using the same candidate generation and notation set as the validator. If exactly one move matches, that move is applied, the move is marked as legal, and the state advances. If no move matches, or more than one move could match, the move is marked as not legal and the reason indicates that the engine could not resolve a single legal interpretation. When the full list is finished, a boolean flag on the object states whether every move in the game was legal under this procedure. The script can run with multiple worker processes and a tunable chunk size so that large files are processed in reasonable time. Output is another JSONL file, for example data/ccpd_labeled.jsonl, where each game still contains the original fields but also includes a list of per-move records with the raw string, a boolean, and a short reason code, plus the whole-game boolean. These labels are defined by the project engine, not by a human tournament report, so they are reproducible and version-controlled with the code.

## How labeled data connects to training

The iterable dataset class in models/dataset.py does not require the extra legality fields to run. It keeps only games that have a non-empty FEN and a standard PGN result among red win, black win, and draw, then simulates the move list to produce tensor encodings of positions and a value target derived from the game result. An optional flag requests a canonical board orientation so that the side to move always appears in a fixed color channel layout, which is a common choice for neural training. The separately labeled file is still valuable: one can filter training to games where the whole-game legality flag is true, or use the per-move flags to study failure modes in the corpus. In that sense, collection and parsing produce the main dataset, while label preprocessing adds an auditable layer that the loader may use or ignore depending on the experiment.

## Closing summary

Dataset collection in this project means reading heterogeneous PGN sources, normalizing them into JSONL with FEN and ordered move strings, and storing encoding and file path for traceability. Label preprocessing means running the same rule engine the game uses in play, normalizing each move token, and writing explicit legal or not-legal decisions plus reasons for every ply. Together, the parse step and the labeling step give a pipeline from raw files to tensor-ready data with optional quality filters backed by engine-grounded labels.
