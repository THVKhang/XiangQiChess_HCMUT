\section{Objectives}

This assignment focuses on building a \textbf{search-based game-playing agent} for \textbf{Xiangqi (Chinese Chess)}.
The implementation targets a large state-space adversarial game and satisfies the project requirements without using machine learning.

Main objectives:
\begin{itemize}
    \item Implement a rule-correct Xiangqi engine (state, moves, legality, terminal conditions).
    \item Implement and compare two search paradigms:
    \begin{itemize}
        \item \textbf{Depth Search:} Minimax
        \item \textbf{Heuristic Search:} Alpha-Beta with evaluation function and move ordering
    \end{itemize}
    \item Provide multiple AI levels (Easy, Medium, Hard).
    \item Support practical evaluation through automated tests and benchmarking scripts.
\end{itemize}

\section{Introduction}

Xiangqi is a deterministic, perfect-information, turn-based zero-sum game with a high branching factor and deep tactical interactions. Exhaustive search is infeasible in practice, so the project applies selective game-tree search with domain heuristics.

The system is implemented in Python with a modular architecture:
\begin{itemize}
    \item \textbf{core/}: rules, board/state representation, legal move generation.
    \item \textbf{agents/}: random agent, minimax/alpha-beta search agents, level wrappers.
    \item \textbf{game/}: backend game loop and match execution.
    \item \textbf{ui/}: Pygame visualization and interaction.
    \item \textbf{tests/}: unit tests for correctness and regression safety.
    \item \textbf{evaluation/}: script-based evaluation and result export.
\end{itemize}

\section{Game Definition and Problem Formulation}

\subsection{Adversarial Search Model}

Xiangqi is modeled as a two-player deterministic zero-sum game:
\[
\langle S, A, T, U \rangle
\]
where:
\begin{itemize}
  \item $S$ is the set of valid game states,
  \item $A(s)$ is the legal action set at state $s$,
  \item $T(s,a)$ is the transition function,
  \item $U(s)$ is utility at terminal states (win/loss/draw).
\end{itemize}

The search agents do not learn from data; they approximate best action by exploring the game tree from current state and applying a handcrafted evaluation at cut-off depth.

\subsection{Rules and Terminal Conditions}

The implementation enforces Xiangqi movement rules for all piece types, including constraints such as palace boundaries, river crossing behavior, horse leg blocking, cannon screen captures, and facing-generals restriction.

Terminal handling includes:
\begin{itemize}
    \item General captured,
    \item Checkmate,
    \item Stalemate,
    \item Threefold repetition draw (to avoid infinite loops).
\end{itemize}

\subsection{State Representation}

A game state includes:
\begin{itemize}
    \item Board piece layout,
    \item Side to move,
    \item Move history.
\end{itemize}

For each search expansion, successor states are generated from legal moves only. This keeps the tree valid by construction and simplifies terminal checks.

\subsection{Core Engine Design Decisions}

To keep the engine robust for both gameplay and search, the implementation follows these decisions:
\begin{itemize}
  \item \textbf{Immutable move objects}: move records are frozen to avoid accidental mutation.
  \item \textbf{Explicit apply/undo path}: search recursion can safely traverse and roll back states.
  \item \textbf{Single legality source}: all agents rely on the same \texttt{legal\_moves} API.
  \item \textbf{Terminal-first handling}: checkmate/general-capture/stalemate/repetition are checked consistently in loop and search utility.
\end{itemize}

This architecture prevents divergence between UI behavior and backend search behavior.

\section{Search Algorithms}

\subsection{Depth Search: Minimax}

Minimax searches the game tree by alternating maximizing and minimizing layers:
\[
V(s) =
\begin{cases}
\max\limits_{a \in A(s)} V(T(s,a)), & \text{if max node} \\
\min\limits_{a \in A(s)} V(T(s,a)), & \text{if min node}
\end{cases}
\]

In this project, Minimax is used as the depth-search baseline and appears explicitly in AI-vs-AI mode for comparison.

\paragraph{How Minimax is implemented in this project}
The implementation follows this sequence:
\begin{enumerate}
  \item Generate all legal moves from current state (using \texttt{legal\_moves} (state)).
  \item Apply one move to create a child state.
  \item Recursively evaluate child states until depth limit.
  \item At leaf level:
  \begin{itemize}
    \item If terminal position: use terminal utility with very large magnitude.
    \item Otherwise: use evaluation function score.
  \end{itemize}
  \item Back-propagate by \texttt{max} at maximizing nodes and \texttt{min} at minimizing nodes.
\end{enumerate}

\paragraph{Complexity perspective}
If branching factor is $b$ and search depth is $d$, plain Minimax complexity is approximately:
\[
O(b^d)
\]
In Xiangqi, this grows quickly; therefore Minimax is mainly used as a depth-search reference and comparison baseline.

\paragraph{Strengths and limitations}
\begin{itemize}
  \item Strength: conceptually simple, deterministic baseline for depth search.
  \item Limitation: explores many unnecessary nodes without pruning, so slower at deeper levels.
\end{itemize}

\subsection{Heuristic Search: Alpha-Beta}

Alpha-Beta pruning reduces the number of expanded nodes while preserving Minimax-optimal decisions:
\begin{itemize}
    \item $\alpha$: best lower bound for maximizing player,
    \item $\beta$: best upper bound for minimizing player,
    \item prune when $\alpha \ge \beta$.
\end{itemize}

Move ordering is used to improve pruning efficiency, especially at medium/hard levels.

  \paragraph{How Alpha-Beta is implemented in this project}
  The engine extends Minimax with three practical optimizations:
  \begin{itemize}
    \item \textbf{Pruning}: skip subtrees when bounds prove they cannot affect final decision.
    \item \textbf{Move ordering}: prioritize tactical moves first (capture priority), improving early cut-offs.
    \item \textbf{Caching}: transposition/evaluation caches reduce repeated computation for equivalent states.
  \end{itemize}

  \paragraph{Practical effect}
  Compared with pure Minimax at the same depth, Alpha-Beta typically searches fewer nodes and returns a move faster. This is why Alpha-Beta is used as the primary heuristic-search agent.

  \paragraph{Complexity perspective}
  Worst-case complexity remains $O(b^d)$, but with effective move ordering the practical explored-node count is significantly reduced. In favorable ordering cases, the effective search depth achievable under fixed time budget is much higher than plain Minimax.

\subsection{Evaluation Function}

The heuristic evaluation combines:
\begin{itemize}
    \item Material values (general, rook, cannon, horse, advisor, elephant, soldier),
    \item Positional adjustments (for example, soldier advancement and piece activity).
\end{itemize}

Terminal utility is assigned a very large magnitude to prioritize forced wins and avoid losing lines:
\[
U_{terminal} = \pm M, \quad M \gg \text{material scale}
\]

\paragraph{Why terminal utility is critical}
Without strong terminal utility, an agent may prefer local material gain over forced checkmate lines. Assigning very large terminal utility ensures tactical correctness: immediate forced win is always prioritized over non-terminal heuristic improvements.

\subsection{Algorithm Assignment in Modes}

\begin{itemize}
  \item \textbf{Human vs AI}: AI side uses search agent with selected level.
  \item \textbf{AI vs Random}: search agent vs random baseline for capability comparison.
  \item \textbf{AI vs AI}: Red uses Alpha-Beta (heuristic search), Black uses Minimax (depth search), with separate side-level selection.
\end{itemize}

This mapping is intentionally chosen to satisfy Assignment 2's requirement of demonstrating both depth search and heuristic search in gameplay.

\subsection{Detailed Test Strategy}

To ensure reliability, testing is split into four layers:
\begin{enumerate}
  \item \textbf{Unit rule tests}: verify local chess rules and movement constraints.
  \item \textbf{State-transition tests}: verify apply/undo and cloning consistency.
  \item \textbf{Integration loop tests}: verify multi-agent interaction and turn execution.
  \item \textbf{Scenario tests}: verify mode-specific behavior and terminal outcomes.
\end{enumerate}

\paragraph{Why this strategy is sufficient}
Xiangqi errors usually come from either (i) illegal move generation, (ii) invalid turn transitions, or (iii) search selecting illegal states. The selected test layers directly target these failure classes.
